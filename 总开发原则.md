# 一、总开发原则

## 1. 不要一上来写业务

错误做法：

```
先写 Controller
→ 随便建 DTO
→ Service 里边查边改
→ Mapper 缺什么补什么
→ 最后发现字段、返回、异常全乱了
```

正确做法：

```
先把规范定死
→ 再把公共能力写好
→ 再按接口一个个纵向打通
```

## 2. 每个接口都按一条完整链路开发

每个接口都要按这个顺序写：

```
DTO
→ Controller
→ Service 接口
→ ServiceImpl
→ Mapper
→ Entity
→ VO
→ Postman 测试
→ 数据库校验
```

不要先把所有 `Controller` 写完，也不要先把所有 `Service` 写完。

## 3. 第一版只做 8 个接口

严格控制范围：

```
1. 新增商品
2. 商品分页查询
3. 修改商品上下架状态
4. 初始化库存
5. 创建订单
6. 取消订单
7. 查询订单详情
8. 查询库存流水
```

第一版不要加：

```
登录
权限
Redis
MQ
支付
秒杀
订单超时取消
操作日志 AOP
复杂后台菜单
```

这些是第二版内容。

# 二、完整标准开发流程

# 阶段 0：阅读接口文档，提取开发清单

你拿到接口文档后，先提取 5 类信息。

## 1. 提取接口清单

写到 `README.md`：

```
POST   /api/v1/products
GET    /api/v1/products
PATCH  /api/v1/products/{productId}/status
POST   /api/v1/stocks/init
POST   /api/v1/orders
POST   /api/v1/orders/{orderNo}/cancel
GET    /api/v1/orders/{orderNo}
GET    /api/v1/stock-flows
```

## 2. 提取统一返回结构

明确项目所有接口都返回：

```
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {},
  "success": true,
  "timestamp": "2026-06-29 15:30:20"
}
```

所以你第一批要写的不是商品接口，而是：

```
ApiResult
ErrorCode
BusinessException
GlobalExceptionHandler
```

## 3. 提取枚举

先整理这些枚举：

```
商品状态：
0-下架
1-上架

订单状态：
10-已创建
20-已取消
30-已完成，第一版预留

库存流水业务类型：
1-下单扣减库存
2-取消订单回滚库存
3-人工初始化库存

操作来源：
0-系统
1-用户
2-管理员
```

这些不要写死在业务代码里到处出现。

## 4. 提取核心业务链路

第一版最核心链路是：

```
商品管理
→ 库存初始化
→ 创建订单
→ 扣减库存
→ 保存订单明细
→ 记录库存流水
→ 取消订单
→ 回滚库存
```

## 5. 提取事务接口

第一版明确需要事务的接口：

```
POST /api/v1/stocks/init
POST /api/v1/orders
POST /api/v1/orders/{orderNo}/cancel
```

这三个接口后面重点验证回滚。

# 阶段 2：建立数据库

## 1. 先建库建表

把之前的 DDL 保存到：

```
src/main/resources/db/schema.sql
```

然后在 MySQL 执行建表。

## 2. 建表后必须检查

执行：

```
SHOW TABLES;
DESC product;
DESC product_stock;
DESC order_info;
DESC order_item;
DESC stock_flow;
```

重点检查每张表是否都有：

```
id
create_time
update_time
is_deleted
version
```

## 3. 不要跳过索引检查

执行：

```
SHOW INDEX FROM product;
SHOW INDEX FROM product_stock;
SHOW INDEX FROM order_info;
SHOW INDEX FROM order_item;
SHOW INDEX FROM stock_flow;
```

重点确认：

```
product.sku_code 唯一
product_stock.product_id 唯一
order_info.order_no 唯一
order_item.order_id + product_id 唯一
stock_flow.biz_no + product_id + biz_type 唯一
```

## 4. 开发习惯

数据库确认没问题后再写 Java 代码。

不要边写代码边改字段名。字段一旦进入实体、Mapper、XML、接口文档，后面改动成本很高。

这个阶段只做“项目能启动、能连库、能返回统一结果”。

## 1. 配置 `application.yml`

第一版至少配置：

```
server.port
spring.datasource.url

spring.datasource.password

mybatis-plus.configuration.map-underscore-to-camel-case

mybatis-plus.global-config.db-config.logic-delete-value
mybatis-plus.global-config.db-config.logic-not-delete-value
```

配置时不要手敲完就继续写业务。启动项目，确认没有报错。

## 2. 启动类

确认启动类位置：

```
com.example.orderinventory.OrderInventoryApplication
```

后续所有包都放在：

```
com.example.orderinventory
```

下面，避免组件扫描不到。

## 3. Mapper 扫描

启动类或配置类上加：

```
@MapperScan("com.example.orderinventory.**.mapper")
```

如果后面报 Mapper 注入失败，第一时间检查这个位置。

# 阶段 4：先写公共层

公共层是整个项目的地基。不要跳过。

## 1. 先写这些类

```
common/result/ApiResult.java
common/result/PageResult.java
common/result/ErrorCode.java
common/exception/BusinessException.java
common/exception/GlobalExceptionHandler.java
common/domain/BaseEntity.java
common/config/MybatisPlusConfig.java
common/handler/MybatisPlusMetaObjectHandler.java
```

## 2. `ApiResult` 负责统一返回

必须支持：

```
成功返回
失败返回
携带 data
不携带 data
```

## 3. `ErrorCode` 负责错误码

把接口文档中的错误码放进去：

```
SUCCESS
PARAM_ERROR
PRODUCT_NOT_FOUND
PRODUCT_STATUS_INVALID
PRODUCT_SKU_DUPLICATE
STOCK_NOT_FOUND
STOCK_ALREADY_EXISTS
STOCK_NOT_ENOUGH
ORDER_NOT_FOUND
ORDER_STATUS_INVALID
ORDER_ITEM_EMPTY
ORDER_ITEM_DUPLICATE
CONCURRENT_UPDATE_FAILED
SYSTEM_ERROR
```

不要在代码里直接写：

```
throw new RuntimeException("商品不存在");
```

应该写：

```
throw new BusinessException(ErrorCode.PRODUCT_NOT_FOUND);
```

## 4. `GlobalExceptionHandler` 负责统一异常

至少处理：

```
BusinessException
MethodArgumentNotValidException
ConstraintViolationException
Exception
```

这一步写完后，先造一个测试接口，确认异常返回结构符合文档。

## 5. `BaseEntity` 负责审计字段

统一放：

```
id
createTime
updateTime
isDeleted
version
```

实体类继承 `BaseEntity`。

# 五、每写一个接口的固定动作

## 1. 先写 DTO

例如新增商品先写：

```
ProductCreateRequest
```

里面只放请求字段：

```
skuCode
productName
productStatus
salePrice
remark
```

不要把数据库审计字段放进 DTO。

## 2. 再写参数校验

DTO 上加校验注解。

例如：

```
不能为空
长度限制
金额大于 0
数量大于 0
```

参数格式校验放 DTO。

业务规则校验放 Service。

## 3. 再写 VO

例如：

```
ProductVO
```

VO 是返回给前端的，不要直接返回 Entity。

## 4. 再写 Controller

Controller 只做三件事：

```
接收参数
触发校验
调用 Service
```

Controller 不写：

```
查数据库
计算金额
判断库存
拼业务流程
try-catch 捕获业务异常
```

## 5. 再写 Service

Service 负责：

```
业务规则
事务边界
调用 Mapper
组装返回结果
```

例如创建订单的所有核心逻辑都在 Service。

## 6. 再写 Mapper

Mapper 只负责数据访问：

```
insert
update
select
分页查询
XML 动态 SQL
```

Mapper 不写业务判断。

## 7. 最后用 Postman 测试

每个接口至少保存：

```
成功用例
参数错误用例
业务异常用例
```

# 八、你要养成的开发习惯

## 1. 先文档，后代码

接口路径、字段、错误码、枚举值，以接口文档为准。

不要边写边改接口。

## 2. 先公共规范，后业务接口

统一返回、统一异常、错误码、分页结构先写。

否则后面每个接口都会写得不一致。

## 3. Controller 永远保持薄

Controller 最好只有 1-3 行核心代码：

```
接收参数
调用 Service
返回 ApiResult
```

看到 Controller 里出现很多 `if`，就要警惕。

## 4. Service 要能讲清业务流程

创建订单接口，你要能画出：

```
参数校验
→ 查商品
→ 查库存
→ 扣库存
→ 保存订单
→ 保存明细
→ 保存流水
→ 提交事务
```

这就是面试表达。

## 5. 写完接口必须看数据库

不要只看接口返回成功。

每次写完接口都要查表：

```
SELECT * FROM product;
SELECT * FROM product_stock;
SELECT * FROM order_info;
SELECT * FROM order_item;
SELECT * FROM stock_flow;
```

尤其是订单接口，必须确认多张表数据一致。

## 6. 失败场景比成功场景更重要

成功场景只能证明“能跑”。

失败场景才能证明“你会做业务”。

重点测：

```
重复 SKU
商品不存在
商品下架
库存不存在
库存不足
重复商品
重复取消订单
事务回滚
```

## 7. 每次只做一个小闭环

不要同时写商品、库存、订单。

正确节奏：

```
写一个接口
测一个接口
提交一次代码
再写下一个接口
```

## 8. 不要把 Entity 当请求对象和返回对象

禁止：

```
Controller 直接接收 Entity
Controller 直接返回 Entity
```

保持：

```
Request DTO：接收请求
Entity：映射数据库
VO：返回前端
```

这是规范项目和普通 CRUD 项目的明显区别。
